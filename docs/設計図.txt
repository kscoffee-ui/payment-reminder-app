今後、実装・UI改善・Codexへの依頼・技術選定・仕様変更を行う際は、必ずこの設計書を参照する。

# 1. プロダクトの目的

カイシュルは、大学生の飲み会・打ち上げ・食事・イベントにおいて、幹事が感じる以下の負担を減らすためのWebアプリである。

- 誰が未払いか管理する面倒さ
- 未払い者へ催促する気まずさ
- 支払い報告を確認する手間
- 大人数イベントで参加者名を最初に全員入力する負担

このアプリの本質は「割り勘計算」ではない。  
本質は、未払いを目立たせ、支払い行動を促し、幹事の未払い回収を助けることである。

決済機能は実装しない。  
支払い自体は、現金・PayPay・銀行振込・その他の外部手段を使う。  
アプリは以下に集中する。

- 支払い案内
- 支払い報告
- 幹事確認
- 未払い可視化
- 催促導線

---

# 2. 基本方針

- React（Vite） + Firebase（Firestore + Hosting）で実装する
- ログイン機能はMVPでは実装しない
- URLを開いたらすぐ使えるUXを最優先にする
- モバイルファーストで実装する
- 主要操作は3タップ以内を目指す
- 決済機能は実装しない
- 幹事画面と参加者画面を分ける
- 参加者は自分で名前を入力してイベントに参加する
- 幹事が最初に全員分の名前を入力する仕様にはしない
- 未払い者の可視化、支払い報告、幹事確認、催促導線を最重要機能として扱う
- 参加者画面に催促機能は表示しない
- イベントロック機能はMVPでは実装しない
- メッセージコピー機能はMVPでは実装しない
- UI改善は、見た目の豪華さよりも「未払いがすぐ分かること」を優先する

## 3-4. TypeScriptの扱い

TypeScriptは長期的には有効だが、現段階では必須ではない。

理由：

- 既存システムがReact + JavaScriptである程度完成している
- 全面TypeScript化は学習コストとリファクタ範囲が大きい
- 今の優先順位は、機能完成・UI改善・実利用検証である

ただし、将来的には以下の型を定義すると保守性が高まる。

```ts
type MemberStatus = "unpaid" | "reported" | "confirmed";
type PaymentMethod = "paypay" | "cash" | "bank" | "other";
```

導入する場合は、新規コンポーネントや `lib` から段階的に行う。

# 4. shadcn/ui導入方針

## 4-1. 導入判断

shadcn/uiは、カイシュルのUIを洗練させるために有効である。  
ただし、全面置き換えは行わず、まずは幹事ダッシュボードだけで試す。

## 4-2. 最初に使うコンポーネント

試験導入で使うshadcn/uiコンポーネントは以下に限定する。

- Button
- Card
- Badge
- Progress
- AlertDialog

## 4-3. 使い方

### Button

主操作を明確にするために使う。

- LINEで催促
- 参加者用URLを共有
- 幹事確認
- 支払いを報告する
- 削除する
- キャンセル

ダッシュボードでは、主CTAは増やしすぎない。  
幹事画面の主CTAは「LINEで催促」を中心にする。

### Card

情報整理に使う。

- イベント概要カード
- 未払いサマリーカード
- 参加者数カード
- 報告済み / 確認待ちカード
- 確認済みカード
- 未払い者リストカード
- 参加者用URL共有カード

最上部では、未払い人数を最も目立たせる。

### Badge

参加者ごとの状態表示に使う。

- 未払い
- 報告済み / 確認待ち
- 確認済み

報告済みは、確認済みと誤認されないようにする。

### Progress

支払い完了率に使う。

- 確認済み人数 / 参加者数 を基準にする
- 報告済み / 確認待ちは完了率に含めない
- 例：16 / 24人、67%

### AlertDialog

危険操作・取り消しにくい操作に限定する。

使う場面：

- 参加者を削除する
- イベントを削除する
- この部屋から抜ける
- 未払いに戻すなど、取り消しにくい操作

基本的に使わない場面：

- 報告済みを確認済みにする
- LINEで催促する
- 参加者用URLを共有する

幹事の操作負担を増やさないことを優先する。

## 4-4. カイシュル専用コンポーネント方針

shadcn/uiを直接各画面で使い回すのではなく、カイシュル専用の共通コンポーネントとして包む。

例：

```txt
src/components/kaishuru/
  StatusBadge.jsx
  SummaryCard.jsx
  MemberCard.jsx
  ReminderButton.jsx
  EventInfoCard.jsx
  ConfirmDeleteDialog.jsx
```

目的：

- 状態表示を統一する
- 色・余白・文言を一元管理する
- 後からデザインを変更しやすくする
- カイシュルらしいUIを作る

# 5. UIデザイン方針

## 5-1. UIの最重要原則

- 未払いを最も目立たせる
- 進捗を上部に表示する
- 支払いCTAを大きくする
- 催促は幹事側に置く
- 参加者の操作はシンプルにする
- モバイルファースト
- 情報はカードで整理する
- 状態は色とラベルで即読できるようにする
- 操作のたびに確認ダイアログを出しすぎない

## 5-2. 色の扱い

色は装飾ではなく、意味を伝えるために使う。

| 役割 | 色の方向性 | 用途 |
|---|---|---|
| Primary | 青 | 通常CTA、リンク、進捗バー、アクティブタブ |
| Unpaid | 赤 | 未払い、注意喚起 |
| Reported | オレンジ | 報告済み / 確認待ち |
| Confirmed | 緑 | 確認済み、完了 |
| LINE | LINE緑 | LINEで催促 |
| Neutral | 白・グレー・黒 | 背景、カード、本文、補助情報 |

推奨カラー例：

```txt
背景：#F8FAFC
カード：#FFFFFF
枠線：#E2E8F0
メイン文字：#0F172A
補助文字：#64748B

メイン青：#2563EB
薄青：#EFF6FF

未払い赤：#DC2626
未払い背景：#FEF2F2

確認待ちオレンジ：#EA580C
確認待ち背景：#FFF7ED

確認済み緑：#16A34A
確認済み背景：#F0FDF4

LINE緑：#06C755
```

## 5-3. 色のルール

- 未払いは赤で強調する
- 報告済み / 確認待ちはオレンジで表示する
- 確認済みは緑で表示する
- 通常操作は青にする
- LINE催促はLINE緑にする
- 削除など危険操作は赤系にする
- 色数を増やしすぎない
- 真っ赤・真緑のベタ塗りを多用しない
- 薄い背景色 + 濃い文字色で状態を表現する

## 5-4. 情報優先順位

幹事ダッシュボードでは以下の順に見せる。

1. イベント概要
2. 未払い人数
3. 支払い完了率
4. 報告済み / 確認待ち人数
5. 確認済み人数
6. 未払い者リスト
7. LINEで催促
8. 参加者用URL共有

# 6. 画面構成

## 6-1. 幹事側：イベント作成画面

入力項目：

- イベント名
- 日付
- 1人あたりの金額
- 支払い方法
  - PayPay
  - 現金
  - 銀行振込
  - その他
- 支払い情報
  - PayPay ID
  - 振込先
  - 現金手渡しの案内
  - その他の支払いメモ
- 任意メモ

重要ルール：

- 参加者名はこの画面では入力しない
- 幹事の入力負担を最小化する
- 金額は整数の円で扱う
- 日付は入力可能にする
- 不正入力時はエラーメッセージを表示する

バリデーション：

- イベント名は必須
- 日付は必須
- 1人あたりの金額は1円以上
- 支払い方法は必須
- 支払い情報は必須

## 6-2. 幹事側：作成完了画面

表示内容：

- イベント作成完了メッセージ
- 参加者用URL
- 幹事用URL
- 参加者用URLの共有導線
- 管理画面へ進むボタン

重要ルール：

- 参加者用URLはLINEグループなどに共有する前提
- 幹事用URLは他人に共有しないよう注意書きを表示する
- URLは推測困難なランダムIDまたはトークンを含める

## 6-3. 幹事側：管理画面（ダッシュボード）

表示内容：

- イベント名
- 日付
- 1人あたりの金額
- 参加者数
- 未払い人数
- 報告済み人数（確認待ち）
- 確認済み人数
- 支払い完了率
- 進捗バー
- 未払い者リスト
- 参加者用URL
- LINEで催促ボタン

重要ルール：

- 未払い者を最も目立たせる
- 報告済みは「幹事確認待ち」と分かるようにする
- 確認済みは安心感のある表示にする
- 参加者側にはこの画面を見せない
- LINE催促は幹事専用にする

## 6-4. 幹事側：参加者一覧画面

表示項目：

- 名前
- 金額
- ステータス
  - 未払い
  - 報告済み（確認待ち）
  - 確認済み
- 支払い方法
- 支払い報告メモ
- 更新日時

操作：

- 参加者を削除する

重要ルール：

- 参加者一覧では、未払い・確認待ち・確認済みの状態確認と絞り込みに集中する
- 参加者一覧では、報告済み（確認待ち）の参加者を確認済みにする操作を表示しない
- 支払い確認は、ダッシュボード右上のベルから開く確認待ち一覧で行う

## 6-5. 幹事側：催促画面（LINE共有）

表示内容：

- 未払い人数
- 未払い者名
- メッセージプレビュー
- LINEで催促ボタン

実装方針：

- メッセージコピー機能は不要
- LINEで催促ボタンを押すと、LINE共有画面を開く
- アプリ側から未払い者へ自動で個別一斉送信する機能は実装しない

## 6-6. 参加者側：参加登録画面

表示内容：

- イベント名
- 日付
- 1人あたりの金額
- 名前入力欄
- 参加するボタン

重要ルール：

- 参加者は自分で名前を入力して参加する
- 幹事が事前に参加者名を登録する必要はない
- 参加後、memberId を localStorage に保存する
- 同じイベントに再アクセスした場合は、自分の支払い画面へ戻す

## 6-7. 参加者側：支払い画面

表示内容：

- 自分の名前
- 支払い金額
- 支払い方法
- 支払い情報
- 任意メモ
- 支払いを報告するボタン
- この部屋から抜けるボタン

重要ルール：

- 参加者には催促画面を表示しない
- 参加者は他人の支払い状態を変更できない
- 決済完了の自動検知はしない
- 支払い後は参加者が「支払いを報告する」を押す
- 報告後はステータスを reported にする
- 幹事が確認すると confirmed になる

## 6-8. 参加者側：報告済み画面（確認待ち）

表示内容：

- 「支払いを報告しました」
- 現在のステータス：確認待ち
- 幹事が確認すると確認済みになる旨
- この部屋から抜けるボタンは表示しない

## 6-9. 参加者側：確認済み画面

表示内容：

- 「支払いが確認されました」
- 現在のステータス：確認済み
- 完了メッセージ
- この部屋から抜けるボタンは表示しない

## 6-10. この部屋から抜ける機能

仕様：

- 参加者側の支払い画面で、本人のステータスが unpaid のときだけ「この部屋から抜ける」ボタンを表示する
- reported / confirmed の参加者は、参加者側から削除できない
- 押すと確認ダイアログを表示し、参加情報が削除されて幹事画面の参加者一覧・未払い一覧からも名前が消えることを明記する
- 確定後、既存の removeMember({ eventId, memberId }) を使って Firestore 上の本人 member ドキュメントを削除する
- Firestore の削除に成功したあと、localStorage に保存された該当イベントの eventId / memberId / memberName を削除する
- 削除成功後、参加登録前の画面へ戻す
- 削除中はボタンを無効化し、「退出中...」と表示して二重送信を防ぐ
- 削除に失敗した場合は参加状態と localStorage を維持し、再試行できるエラーを表示する

注意：

- Firestore のデータ構造は変更しない
- 参加者側から削除できるのは、参加者用URLで開いている本人の unpaid member のみとする

# 7. ステータス設計

参加者の支払い状態は以下の3段階で管理する。

## unpaid

意味：まだ支払い報告がされていない状態。

表示：

- 未払い
- 赤系で強調
- 幹事画面では上部に表示

## reported

意味：参加者が支払いを報告したが、幹事がまだ確認していない状態。

表示：

- 報告済み
- 確認待ち
- オレンジ系で表示

## confirmed

意味：幹事が支払いを確認済みの状態。

表示：

- 確認済み
- 緑系で表示

# 9. URL設計

## 参加者用URL

```txt
/join/:eventId?token=:participantToken
```

用途：

- 参加者がイベントに参加する
- 参加者が支払い情報を確認する
- 参加者が支払い報告する

## 幹事用URL

```txt
/admin/:eventId?token=:adminToken
```

用途：

- 幹事が管理画面を見る
- 報告済みを確認済みにする
- 参加者を削除する
- LINEで催促する

重要ルール：

- participantToken と adminToken は別にする
- adminToken を知らない参加者は管理画面に入れない
- ログインなしのため、トークン付きURLで最低限の権限制御を行う

# 10. localStorage設計

参加者側ではイベントごとに以下を保存する。

```js
{
  eventId: string,
  memberId: string,
  memberName: string
}
```

キー例：

```txt
paymentReminderMember:{eventId}
```

「この部屋から抜ける」を押した場合：

- Firestore 上の本人 member ドキュメント削除に成功したあと、該当キーを削除する
- 参加登録画面に戻す
- 削除に失敗した場合は該当キーを残し、参加状態を維持する

幹事側については、ログインなしMVPの補助として、作成済みイベント情報をlocalStorageに保存し、トップ画面から最近作成したイベントへ戻れる導線を設ける。

# 11. リアルタイム同期とFirebaseクォータ対策

Firestore の onSnapshot を使う。

## 11-1. 幹事画面

- event を購読する
- members サブコレクションを購読する
- 参加者数、未払い数、報告済み数、確認済み数、進捗率をリアルタイムで更新する

## 11-2. 参加者画面

- event を必要最小限で読む
- 自分の member ドキュメントのみ購読する
- 参加者側で members 全体を購読しない
- 幹事が confirmed にしたら、確認済み画面に切り替える

## 11-3. クォータ対策

重要ルール：

- 参加者側は自分の member だけ購読する
- members 全体を読むのは幹事画面だけにする
- 不要な画面では onSnapshot を使わない
- useEffect の cleanup で必ず unsubscribe する
- ポーリングを多用しない
- 非表示タブや不要画面で購読し続けない
- Firebase Usage を確認し、reads / writes を観測する

40人規模の飲み会でも、参加者側が自分のmemberだけを読む設計であれば、無料枠の範囲で小規模テストは可能な見込み。  
ただし、参加者全員に members 全体を購読させる設計は避ける。

# 12. 計算ルール

イベント作成時に「1人あたりの金額」を入力する方式を基本とする。

理由：

- 大人数イベントで総額と人数が確定していないケースに対応しやすい
- 幹事が参加者名を入力しないため、総額から均等割りするより自然
- 打ち上げや飲み会では「1人3,000円」のような会費制が多い

金額は必ず整数の円で扱う。  
浮動小数点は使用しない。

支払い完了率は以下で計算する。

```txt
確認済み人数 / 参加者数
```

報告済み / 確認待ちは完了率に含めない。

- reported の参加者のみ confirmed にできる

# 14. 通信・エラー処理

- データ取得中はローディング表示
- エラー時は明示的に表示
- 更新中はボタンを無効化する
- 二重送信を防止する
- URLトークンが不正な場合はエラー画面を表示する
- Firestoreエラー時はユーザーが状況を理解できる文言にする
- Firebaseクォータ系エラーが出た場合は再試行を抑制する

---

# 15. セキュリティ方針

- ログインはMVPでは実装しない
- 参加者用URLと幹事用URLを分ける
- participantToken と adminToken を分ける
- URLを知っている人だけがアクセスできる最低限の権限制御にする
- 参加者は自分のmember以外を変更できない
- 幹事のみが参加者一覧・削除・確認済み操作を行える
- 本格的な認証・決済・自動送金は実装しない
- Firestore Security Rules を必ず確認する

---

# 16. ドメイン・公開方針

独自ドメインの取得は可能。  
アプリ名は「カイシュル」で確定する。

推奨構成：

```txt
アプリ本体：Firebase Hosting
データ：Firestore
独自ドメイン：Firebase Hostingに接続
カラフルボックス：LP、使い方ページ、紹介ページ、問い合わせページ用
```

例：

```txt
kaishuru.com
app.kaishuru.com
```

または初期段階では：

```txt
kaishuru.com
```

をアプリ本体に接続してもよい。

公開前に確認すること：

- Firestore Security Rules
- participantToken / adminToken の分離
- 参加者側がmembers全体を読んでいないこと
- スマホ表示
- エラー表示
- ローディング表示
- 最低限の使い方説明
- 問い合わせ先
- 簡易プライバシーポリシー

# 17. 収益化方針

MVPでは収益化しない。

理由：

- カイシュルの強みは、ログインなし・URLですぐ使えること
- 広告や課金を初期導入すると、参加者の離脱につながる
- 参加者側に課金や広告を強く出すと、本質である支払い報告の邪魔になる
- まずは利用検証を優先する

将来的に検討する収益化：

- 幹事向けプレミアム
- サークル・団体向けプラン
- 過去イベント管理
- 複数イベント管理
- メンバーリスト保存
- CSV出力
- 催促テンプレート
- 回収レポート

広告は補助的な選択肢であり、メイン収益にはしない。  
決済手数料モデルはMVPでは採用しない。

最初に検証する指標：

- 幹事がイベントを作成するか
- 参加者がURLから登録するか
- 支払い報告まで進むか
- 幹事が確認済みにするか
- LINE催促を使うか
- 同じ幹事が2回目も使うか

# 21. 最重要ルール

カイシュルの本質は「割り勘」ではなく「未払い回収」である。

そのため、以下を常に優先する。

- 幹事が参加者名を入力しなくていい
- 参加者が自分で登録できる
- 未払い者がすぐ分かる
- 支払い報告と幹事確認ができる
- 催促は幹事専用にする
- 参加者の操作はシンプルにする
- 幹事の心理的負担を減らす
- 未払い者に支払い行動を促す
- 技術導入は目的ではなく、未払い回収UXを良くするための手段とする
