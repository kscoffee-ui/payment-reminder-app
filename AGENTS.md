## 1. プロジェクト概要

カイシュルは、大学生向けの未払い回収 Web アプリです。

目的は、飲み会・打ち上げ・イベントなどで幹事が感じる以下の負担を減らすことです。

- 誰が未払いか管理する面倒さ
- 未払い者へ催促する気まずさ
- 支払い報告を確認する手間
- 大人数イベントで参加者名や金額を管理する負担

このアプリの本質は「割り勘計算」ではなく「未払い回収」です。
そのため、実装・UI改善・仕様判断では常に以下を優先してください。

- 未払い者を目立たせる
- 支払い行動を促す
- 幹事の心理的負担を減らす
- 支払い報告と確認作業を分かりやすくする
- スマホで迷わず操作できる軽い UI にする

## 3. 技術構成

基本構成は以下です。

- React
- Vite
- Firebase Firestore
- Firebase Hosting または Vercel
- モバイルファースト
- ログイン不要 UX を優先
- 将来的に PWA 化を検討

MVP では以下を実装しません。

- 決済機能
- 自動送金
- 本格的なログイン機能
- ユーザー向けテーマ変更機能
- 複雑な管理画面
- 汎用ノーコードビルダー

---

## 4. 最重要 UX ルール

カイシュルでは、未払い回収に直結する情報を最優先します。

- 未払い人数を最も目立たせる
- 未払いメンバーを見つけやすくする
- 報告済み / 確認待ちは、確認済みと誤認されないようにする
- 確認済み操作は幹事側だけに置く
- 催促導線は幹事側だけに置く
- 参加者側には催促機能を表示しない
- 参加者側の操作は「支払い報告」を中心にシンプルにする
- 主要操作はなるべく 3 タップ以内で完了させる
- 操作のたびに確認ダイアログを出しすぎない

## 5. UI デザイン方針

UI はスマホファーストで、iOS 風の軽い見た目を優先します。

### 色の役割

色は装飾ではなく、状態を伝えるために使います。

- Primary: 青。通常 CTA、リンク、進捗バー、アクティブタブ
- Unpaid: 赤。未払い、注意喚起
- Reported: オレンジ。報告済み / 確認待ち
- Confirmed: 緑。確認済み、完了
- LINE: LINE 緑。LINE で催促
- Neutral: 白・グレー・黒。背景、カード、本文、補助情報

### 推奨カラー

```txt
背景：#F8FAFC
カード：#FFFFFF
枠線：#E2E8F0
メイン文字：#0F172A
補助文字：#64748B

メイン青：#2563EB
薄青：#EFF6FF

未払い赤：#DC2626
未払い背景：#FEF2F2

確認待ちオレンジ：#EA580C
確認待ち背景：#FFF7ED

確認済み緑：#16A34A
確認済み背景：#F0FDF4

LINE緑：#06C755
```

## 6. コンポーネント方針

shadcn/ui を直接各画面に散らばらせず、カイシュル専用の共通コンポーネントで包んでください。

例：

```txt
src/components/kaishuru/
  KaishuruButton.jsx
  KaishuruCard.jsx
  KaishuruInput.jsx
  StatusBadge.jsx
  SummaryCard.jsx
  MemberCard.jsx
  MemberRow.jsx
  ReminderButton.jsx
  EventInfoCard.jsx
  ConfirmDeleteDialog.jsx
```

目的：

- 状態表示を統一する
- 色・余白・文言を一元管理する
- 後からデザインを変更しやすくする
- カイシュルらしい UI を保つ

### AlertDialog / ConfirmDialog の扱い

危険操作・取り消しにくい操作に限定してください。

使う場面：

- 参加者を削除する
- イベントを削除する
- この部屋から抜ける
- 未払いに戻すなど、取り消しにくい操作

基本的に使わない場面：

- 報告済みを確認済みにする
- LINE で催促する
- 参加者用 URL を共有する

## 7. Firestore / Firebase ルール

指示がない限り、以下には触らないでください。

- Firebase 設定
- Firebase 環境変数
- Firestore のデータ構造
- Firestore Security Rules
- 既存の読み書き処理
- token / URL 設計

MVP の権限制御方針：

- ログインは MVP では実装しない
- 参加者用 URL と幹事用 URL を分ける
- `participantToken` と `adminToken` を分ける
- URL を知っている人だけがアクセスできる最低限の権限制御にする
- 参加者は自分の member 以外を変更できない
- 幹事のみが参加者一覧・削除・確認済み操作を行える
- 本格的な認証・決済・自動送金は実装しない

Firestore に触る必要がある場合は、作業前に変更理由と影響範囲を明確にしてください。

## 8. 開発作業の基本ルール

作業前に必ず確認してください。

```bash
git status
git branch
```

作業は main に直接行わず、原則として feature ブランチで行ってください。

例：

```bash
git switch -c feature/kaishuru-ui-studio-mvp
```

実装時のルール：

- いきなり大規模変更しない
- 1 回の作業範囲はできるだけ 3〜5 ファイル程度に抑える
- 指定された作業範囲だけを変更する
- ついでの仕様追加をしない
- UI 変更とロジック変更を混ぜすぎない
- 既存 MVP 機能を壊さない
- 初心者でも追えるように、必要な箇所には短いコメントを入れる

## 11. Kaishuru UI Studio ルール

Kaishuru UI Studio は、開発者がカイシュル本体の UI を調整するための開発用ツールです。
ユーザー向け機能ではありません。

目的：

- 色、角丸、余白、影、文字サイズなどを画面上で調整する
- Button / Card / Input / StatusBadge / EventCard / MemberRow などを確認する
- Dashboard / Members / PaymentReport などのスマホプレビューを確認する
- 最終的にカイシュル本体の UI に反映しやすくする

MVP でやること：

- `/dev/studio` を作る
- 左に編集パネル、右にスマホプレビューを置く
- localStorage に保存する
- Reset できるようにする
- JSON としてコピーできるようにする
- CSS 変数に反映する
- 本番環境では表示しない

MVP でやらないこと：

- 汎用ノーコードツール化
- ドラッグ＆ドロップによる完全な自由配置
- Firestore 連携
- ログイン付きテーマ保存
- ユーザー向けテーマ変更
- AI 自動デザイン生成
- Figma 完全再現

UI Studio を作る場合も、本体で使う共通コンポーネントを再利用してください。
Studio 専用の見た目を作り込みすぎないでください。

## 12. Codex のトークン節約ルール

Codex に一度で大きな作業をさせないでください。

悪い例：

```txt
UI Studio を全部作ってください。
```

良い例：

```txt
まずテーマ基盤だけ作ってください。
```

```txt
次に StatusBadge と KaishuruButton だけ CSS 変数対応してください。
```

```txt
次に /dev/studio の画面枠だけ作ってください。
```

```txt
次に DashboardPreview だけ作ってください。
```

小さく分けることで、トークン消費、失敗、修正コストを減らします。

## 13. 仕様追加の判断基準

新しい機能や UI を追加する前に、以下を確認してください。

- 未払い回収率を上げるか
- 幹事の負担を減らすか
- 参加者の操作を重くしないか
- Firestore の読み取り・書き込みを増やしすぎないか
- MVP の範囲を超えていないか
- 設計図に反していないか

上記に当てはまらない機能は、原則として追加しないでください。
