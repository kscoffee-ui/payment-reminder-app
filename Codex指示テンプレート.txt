# Codex指示テンプレート.txt

このファイルは、ChatGPTでCodex向けの実装プロンプトを作成するときのルールです。
目的は、Codexのトークン消費を抑えつつ、カイシュル開発の方針からズレない指示を作ることです。

---

## 1. 基本方針

Codexに投げるプロンプトは、毎回長くしすぎない。
固定ルールは docs/設計図.txt と AGENTS.md に任せ、Codexへの指示では「今回やること」だけを短く具体的に書く。

カイシュルの本質は、割り勘ではなく未払い回収である。
実装では常に以下を優先する。

* 未払い者を目立たせる
* 支払い報告を分かりやすくする
* 幹事の確認作業を軽くする
* 催促の心理的負担を減らす
* スマホで3タップ以内に主要操作が終わるUIを目指す
* 勝手な機能追加や複雑化を避ける

---

## 2. Codexに毎回必ず伝えること

Codexへの指示には、基本的に以下を入れる。

```txt
AGENTS.md と docs/設計図.txt を確認してください。
今回の作業範囲だけを実装してください。
Firestore、Firebase設定、環境変数には指示がない限り触らないでください。
変更後は npm run build を実行し、git diff と git status を確認してください。
```

---

## 3. Codexプロンプトの基本形

```txt
AGENTS.md と docs/設計図.txt を確認してください。

今回の目的：
【ここに今回の目的を1つだけ書く】

作業範囲：
- 【作業1】
- 【作業2】
- 【作業3】

禁止事項：
- 指定範囲外を大きく変更しない
- Firestore処理に触らない
- Firebase設定や環境変数に触らない
- 既存仕様を勝手に増やさない
- デザインや機能を複雑化しない

完了条件：
- npm run build が通る
- git diff で変更内容を確認できる
- git status を確認する
- 変更したファイル、実装内容、残課題を報告する
```

---

## 4. 調査だけさせるときのテンプレート

実装前に状況確認だけしたい場合は、以下を使う。

```txt
AGENTS.md と docs/設計図.txt を確認してください。

今回はまだコードを変更しないでください。
現在のファイル構成と関連ファイルを確認し、【実装したい内容】を追加する場合に変更が必要なファイルだけを一覧化してください。

出力してほしい内容：
- 関連ファイル
- 変更が必要なファイル
- 新規作成が必要なファイル
- 実装の順番
- 注意点

まだ実装はしないでください。
```

- Firebase設定や環境変数に触らない

完了条件：
- npm run build が通る

- git status を確認する

## 6. UI Studio用テンプレート

```txt
AGENTS.md と docs/設計図.txt を確認してください。

今回の目的：
Kaishuru UI Studio MVPのために、【今回の対象】だけを実装してください。

作業範囲：
- 【例：src/theme/defaultTheme.js を作成】
- 【例：src/theme/applyTheme.js を作成】
- 【例：CSS変数に反映できる形にする】

禁止事項：
- Firestore処理に触らない
- Firebase設定や環境変数に触らない
- /dev/studio 全体を一気に作り込まない
- ドラッグ＆ドロップや汎用ノーコード機能は追加しない
- ユーザー向けのテーマ変更機能は作らない
- 本体仕様を勝手に増やさない

完了条件：
- npm run build が通る
- 変更ファイルを報告する
- 次に実装すべき小さな作業を提案する
```

## 7. バグ修正用テンプレート

```txt
AGENTS.md と docs/設計図.txt を確認してください。

現在の問題：
【起きている不具合を書く】

再現手順：
1. 【手順1】
2. 【手順2】
3. 【手順3】

期待する動作：
【本来どうなってほしいか】

作業範囲：
まず原因を調査してください。
原因が分かったら、最小変更で修正してください。

禁止事項：
- 関係ないUIやロジックを変更しない
- Firestore処理に不要な変更を加えない
- Firebase設定や環境変数に触らない
- 大規模リファクタリングをしない

完了条件：
- npm run build が通る
- 修正原因を説明する
- 変更ファイルを報告する
- git diff と git status を確認する
```

## 8. リファクタリング用テンプレート

```txt
AGENTS.md と docs/設計図.txt を確認してください。

今回の目的：
【対象ファイル / 対象処理】を、仕様を変えずに整理してください。

作業範囲：
- 【対象ファイル1】
- 【対象ファイル2】

守ること：
- 見た目と挙動を変えない
- Firestoreのデータ構造を変えない
- 仕様を追加しない
- 変更理由が分かる範囲に留める

完了条件：
- npm run build が通る
- 変更前後で挙動が変わっていないことを説明する
- 変更ファイルを報告する
```

## 9. Codexに投げる前のチェック

Codexに投げる前に、毎回以下を確認する。

* 今回の目的は1つに絞れているか
* 作業範囲が3〜5ファイル程度に収まっているか
* Firestoreに触る必要が本当にあるか
* UI変更なのか、ロジック変更なのかを分けているか
* 「ついでにこれも」は入っていないか
* 完了条件が明確か
* build確認を入れているか

---

## 10. Codexのトークン節約ルール

Codexに一度で大きな作業をさせない。
以下のように分割する。

悪い例：

```txt
UI Studioを全部作ってください。
```

良い例：

```txt
まずテーマ基盤だけ作ってください。
```

```txt
次にStatusBadgeとKaishuruButtonだけCSS変数対応してください。
```

```txt
次に/dev/studioの画面枠だけ作ってください。
```

```txt
次にDashboardPreviewだけ作ってください。
```

小さく分けることで、トークン消費、失敗、修正コストを減らす。

---

## 11. 作業後にCodexへ必ず報告させる内容

```txt
最後に以下を報告してください。

- 参照したファイル
- 追加・変更したファイル
- 実装した内容
- 実行した検証コマンド
- npm run build の結果
- Firestore / Firebase設定に触ったかどうか
- 残っている課題
- 次に進めるべき作業
```

---

## 12. カイシュル開発で避けること

以下はCodexに実装させない。

* 決済機能
* ログイン必須化
* ユーザー向けテーマ変更
* 複雑な管理画面
* 汎用ノーコードビルダー
* AI自動デザイン生成
* Firestore構造の勝手な変更
* 参加者側に催促機能を出すこと
* 報告済みと確認済みが紛らわしいUI
* 幹事の操作が増える過剰な確認ダイアログ

---

## 13. 最短プロンプト例

小さい作業なら、以下くらい短くてよい。

```txt
AGENTS.md と docs/設計図.txt を確認してください。

今回は src/theme/defaultTheme.js と src/theme/applyTheme.js だけ作成してください。
Kaishuru UI Studio MVPで使うテーマ基盤です。

Firestore、Firebase設定、App.jsxには触らないでください。
変更後は npm run build、git diff、git status を確認し、変更内容と残課題を報告してください。
```
